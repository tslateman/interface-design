Beyond working correctly, good code is a joy to read, modify, and build upon.

The Essence of Good Code

Good code is like good writing: it communicates clearly, respects the reader's time, and accomplishes its purpose elegantly. It's not about showing off clever tricks or cramming everything into one line‚Äîit's about empathy for the next developer who reads it (often yourself in six months).

The Three Audiences of Code

  1. 1. The Computer: It must work correctly and efficiently
  2. 2. Other Developers: It must be understandable and maintainable
  3. 3. Future You: It must be clear enough that you can understand it months later

Good code optimizes for all three, but prioritizes human understanding.

Core Principles of Good Code

1. Clarity Above Cleverness

Code is read far more often than it's written. Optimize for readability.

‚ùå Clever but Unclear

// What does this do?
const r = u.map(x => x.a ? x.b : x.c).filter(Boolean).reduce((a,b) => a+b, 0);

// Nested ternary madness
const status = user.isActive ? user.isPremium ? 'premium' : 'basic' : 'inactive';

‚úÖ Clear and Obvious

// Calculate total of valid amounts
const validAmounts = users
  .map(user => user.isValid ? user.amount : user.defaultAmount)
  .filter(amount => amount != null);
const total = validAmounts.reduce((sum, amount) => sum + amount, 0);

// Clear status determination
function getUserStatus(user) {
  if (!user.isActive) return 'inactive';
  if (user.isPremium) return 'premium';
  return 'basic';
}

2. Simplicity Through Focus

Each piece of code should do one thing well. Complexity emerges from composition, not complication.

‚ùå Doing Too Much

# Function does multiple unrelated things
def process_user_data(users):
    # Validate users
    valid_users = []
    for user in users:
        if user.age > 18 and user.email:
            valid_users.append(user)
    
    # Calculate statistics
    total_age = sum(u.age for u in valid_users)
    avg_age = total_age / len(valid_users)
    
    # Send emails
    for user in valid_users:
        send_welcome_email(user.email)
    
    # Save to database
    db.save_users(valid_users)
    
    return avg_age

‚úÖ Single Responsibility

# Each function has one clear purpose
def validate_users(users):
    return [u for u in users if u.age > 18 and u.email]

def calculate_average_age(users):
    if not users:
        return 0
    return sum(u.age for u in users) / len(users)

def send_welcome_emails(users):
    for user in users:
        send_welcome_email(user.email)

# Compose them as needed
valid_users = validate_users(users)
avg_age = calculate_average_age(valid_users)
send_welcome_emails(valid_users)
db.save_users(valid_users)

3. Consistency is Kindness

Predictable patterns reduce cognitive load. Pick conventions and stick to them.

üìã Consistency Checklist

  • ‚úì Naming: camelCase or snake_case, not both
  • ‚úì Error Handling: Always the same pattern (exceptions, callbacks, or promises)
  • ‚úì File Structure: Predictable organization
  • ‚úì Code Style: Consistent formatting (use a linter!)
  • ‚úì Patterns: Similar problems solved similarly

4. Defensive but Not Paranoid

Handle edge cases gracefully without overcomplicating the happy path.

// ‚úÖ Good defensive coding
function calculateDiscount(price, discountPercent) {
  // Validate inputs
  if (typeof price !== 'number' || price < 0) {
    throw new Error('Price must be a positive number');
  }
  
  if (typeof discountPercent !== 'number' || discountPercent < 0 || discountPercent > 100) {
    throw new Error('Discount must be between 0 and 100');
  }
  
  // Simple, clear calculation
  return price * (1 - discountPercent / 100);
}

Error Handling That Helps

‚ùå Unhelpful Errors

function connectToDatabase(config) {
  if (!config) {
    throw new Error('Error');
  }
  
  if (!config.host || !config.port) {
    return null; // Silent failure
  }
  
  try {
    return connect(config);
  } catch (e) {
    console.log('Failed'); // Swallows details
    return false;
  }
}

‚úÖ Helpful Error Messages

function connectToDatabase(config) {
  if (!config) {
    throw new Error('Database config is required');
  }
  
  if (!config.host) {
    throw new Error(`Database host is required, received: ${JSON.stringify(config)}`);
  }
  
  if (!config.port) {
    throw new Error(`Database port is required, received: ${JSON.stringify(config)}`);
  }
  
  try {
    return connect(config);
  } catch (error) {
    throw new Error(`Failed to connect to ${config.host}:${config.port} - ${error.message}`);
  }
}

Code Smells to Avoid

üö® Long Methods

If you need to scroll to see the whole function, it's doing too much.

Fix: Extract smaller functions with clear names

üö® Deep Nesting

Multiple levels of indentation make code hard to follow.

Fix: Early returns, extract functions, simplify logic

üö® Magic Numbers

Unexplained numbers scattered throughout code.

Fix: Named constants with clear meaning

üö® Duplicated Code

Same logic repeated in multiple places.

Fix: Extract shared functionality

üö® Dead Code

Commented out code, unused variables, unreachable branches.

Fix: Delete it (version control remembers)

üö® Premature Optimization

Complex code for hypothetical performance gains.

Fix: Write clear code first, optimize with data

The Test of Time

Good code ages well. It remains understandable and modifiable long after it was written. Here's how to write code that stands the test of time:

Write for Change

  • ‚Ä¢ Assume requirements will evolve
  • ‚Ä¢ Make the easy changes easy
  • ‚Ä¢ Isolate the parts likely to change
  • ‚Ä¢ Use configuration for values that might vary

Document the "Why"

  • ‚Ä¢ Code shows what and how, comments explain why
  • ‚Ä¢ Document non-obvious decisions
  • ‚Ä¢ Explain trade-offs you made
  • ‚Ä¢ Link to relevant discussions or tickets

Make It Testable

  • ‚Ä¢ Pure functions are easier to test
  • ‚Ä¢ Inject dependencies rather than hardcoding
  • ‚Ä¢ Separate business logic from I/O
  • ‚Ä¢ Tests document expected behavior

Quick Reference Card

‚úÖ DO

  • ‚úì Write code for humans first
  • ‚úì Keep functions small and focused
  • ‚úì Use descriptive names
  • ‚úì Handle errors explicitly
  • ‚úì Be consistent with patterns
  • ‚úì Refactor when you understand better
  • ‚úì Write tests for complex logic
  • ‚úì Document surprising decisions

‚ùå DON'T

  • ‚úó Optimize prematurely
  • ‚úó Use clever one-liners
  • ‚úó Copy-paste code
  • ‚úó Ignore error cases
  • ‚úó Mix concerns in one function
  • ‚úó Leave commented-out code
  • ‚úó Use magic numbers
  • ‚úó Assume context is obvious

Final Thoughts

Writing good code is a craft that improves with practice and feedback. It's not about perfection‚Äîit's about continuous improvement and empathy for others (including future you).

Remember:

  • ‚Ä¢ Good code is honest - It does what it says, no surprises
  • ‚Ä¢ Good code is humble - It doesn't try to be clever
  • ‚Ä¢ Good code is kind - It helps the next developer succeed
  • ‚Ä¢ Good code evolves - It's never done, just good enough for now

The best code is not the code that impresses other developers.
It's the code that doesn't need to.
